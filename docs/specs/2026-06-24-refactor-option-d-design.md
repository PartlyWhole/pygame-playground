# Refactor Option D — engine-extraction design (THINK-ONLY; for review)

> Concrete design for extracting the cooperative engine into a hand-authored `vendor/engine.mjs` (no build step,
> automerge-collab precedent). Execution gated on the File-Explorer cluster merging; sequence = Slice B→C→refactor-D→lesson-UI.

> **POST-IMPLEMENTATION CORRECTIONS (refactor-D shipped 2026-06-24, merged `5388b39`):**
> 1. This doc says **`window.pyodide`** throughout — that is INACCURATE. Tests reach the interpreter by the
>    **bare-name `pyodide`** global (the host inline `<script>` is classic, so its top-level `let pyodide`
>    is reachable in `page.evaluate`); `window.pyodide` is never used. Never convert the host `<script>` to
>    `type="module"`; always assign the booted instance to the host's top-level `let pyodide`.
> 2. `engine.boot(deps)` required one dep beyond this design's list: **`setPyodide(inst)`** — the engine
>    publishes the instance to the host's `pyodide` *immediately* after `loadPyodide()`, before `hydrateAssets`/
>    `_memfs` (which guard `if (!pyodide)`) read it. Without it, asset hydration no-ops mid-boot.
>
> The as-built record is `docs/superpowers/plans/2026-06-24-refactor-d-engine-extraction.md`.

## Engine public surface

**Exports (`vendor/engine.mjs`):**
- BOOT_PY (String.raw Python source const, re-exported verbatim — the cooperative single-file engine: _state, _pause_gate, _Clock, __yield__/__sleep__, the AST transform passes _Asyncify/_Awaiter/_InjectYield/_SyncBarrier/_time_names/_is_gameloop/_shallow, _transform, _run/_start/_stop/_pause/_resume)
- PROJECT_PY (String.raw Python source const, re-exported verbatim — the multi-file engine: __maybe_await__, _AwaitCalls, _AsyncifyCoop, _ModuleCoop, _check_loop_placement, _CoopCallCheck, _transform_module/_transform_entry, the _CoopPathFinder/_CoopLoader MetaPathFinder, _install_finder, writePath, _ensure_init_py/_prune_empty_dirs/_dotted_names_for, _run_project/_start_project/_purge_project_files)
- createEngine(deps) factory returning an engine object — the ONLY new abstraction; everything below is a method on it
- engine.boot() — async: runs the host-supplied Pyodide loader, registers the canvas, runs BOOT_PY then PROJECT_PY via pyodide.runPythonAsync, hydrates assets; returns the booted promise. Installs the Python globals into pyodide.globals EXACTLY as boot() does today.
- engine.start(src) — wraps pyodide.globals.get('_start')(src); returns the task. Single-file path.
- engine.startProject(filesObj, entry) — wraps _start_project; does the toPy(files)/destroy() dance; returns the task.
- engine.stop() — pyodide.runPython('_stop()')
- engine.pause() / engine.resume() — pyodide.runPython('_pause()'/'_resume()')
- engine.isPaused() — bool(_state.get('paused'))
- engine.purgeProjectFiles() — pyodide.runPython("_state['via_project']=False; _purge_project_files()") for the single-file reset path
- engine.runPython / engine.pyodide passthrough (or just keep window.pyodide as today) so tests + lazy loaders that poke _state directly keep working
- OPTIONAL (Phase-2 only, NOT this extraction): stepN(n)/stateSnapshot()/taskDone() headless helpers

**Injected by the host (`index.html`):**
- loadPyodide: an async () => pyodide factory (host owns the PYODIDE_BASE CDN URL + import(PYODIDE_BASE+'pyodide.mjs') + mod.loadPyodide({indexURL}) — keeps the CDN URL and version pin in index.html, not the module)
- canvas: the #canvas DOM element (host passes canvasEl; engine does the pyodide.canvas.setCanvas2D(canvas) / _module.canvas fallback + the SDL_EMSCRIPTEN_KEYBOARD_ELEMENT runPython)
- setStatus: (cls, token) => void — the host's single #status writer, so the engine can drive 'loading Python…'/'loading pygame…' during boot WITHOUT knowing the DOM (token strings still owned/asserted by host)
- logSink: { out:(s)=>void, err:(s)=>void } — wired to pyodide.setStdout/setStderr batched; host passes logLine bound to its #console
- hydrateAssets: async () => void — host's assetFS.hydrateAll, called at end of boot (engine stays asset-storage-agnostic; it only needs MEMFS populated before first run)
- fileSource is NOT injected as a dep — run() reads project.serialize().files / editor.getValue() in the HOST and passes the snapshot as a plain string/dict argument to start()/startProject(); the engine never imports the project model

**Notes:** The engine stays UI-agnostic: it knows pyodide, a canvas element, a status-token callback, a log sink, and plain string/dict program snapshots. It never imports project, renderTabs, collab, assetFS, or any DOM id. CRITICAL CONSTRAINT discovered in test/runmodel.mjs, spike-runstop.mjs, spike-pause.mjs, multifile.mjs: tests poke the PYODIDE PYTHON NAMESPACE directly (pyodide.runPython('_stop()'), _state['task'], _state['via_project'], id(_state['task'])) and use the bare global window.pyodide. Therefore the extraction must NOT move the Python globals off pyodide.globals — after engine.boot() the Python namespace must be byte-identical to today (same _start/_stop/_pause/_resume/_state/__yield__ in pyodide.globals). The engine.mjs is a thin JS wrapper over the SAME runPythonAsync(BOOT_PY)+runPythonAsync(PROJECT_PY); only the source strings and the boot orchestration relocate. window.pyodide MUST remain set so the ~40 direct test pokes keep resolving.

## Python source handling
BOOT_PY and PROJECT_PY move into vendor/engine.mjs as `export const BOOT_PY = String.raw\`...\`` / `export const PROJECT_PY = String.raw\`...\``, copied byte-for-byte from index.html lines 859-1096 and 1105-1424. String.raw is retained (it stops backslash-escape processing and lets the brace-heavy Python pass through). The single hazard is JS `${}` interpolation, which String.raw still honors — verified the current Python source contains zero `${` sequences (only bare `{}` dict/set literals, which are inert under String.raw), so a verbatim move is safe. Do NOT reformat/re-indent the Python and do NOT externalize to a fetched `.py` file (would add a network round-trip + a new first-run failure mode). engine.boot() executes them in strict order via `await pyodide.runPythonAsync(BOOT_PY); await pyodide.runPythonAsync(PROJECT_PY)` because PROJECT_PY depends on names BOOT_PY defines. Acceptance check: stripping the `const`→`export const` prefix, a diff of the two strings against the originals is empty.

## Load mechanism
Dynamic import() on a cached-promise gate, identical to the Automerge model: `import(new URL('./vendor/engine.mjs', document.baseURI).href)`. NOT a static top-level import and NOT <script type=module> (both would eager-load the module graph at first paint). The relative+document.baseURI form is load-bearing for the GitHub Pages project subpath. createEngine(deps) is called once and cached; engine.boot() chains a single booted promise so concurrent Run clicks share one boot.

## First-paint impact
Zero first-paint regression is the requirement and is achievable. Note today's boot is ALREADY eager (line 3378 `const booted = boot()` fires at module-eval, so Pyodide warms on page load, not on first Run). Recommended: keep that exact behavior (behavior A) — index.html does `const booted = loadEngine().then(e => e.boot())`, dynamically importing engine.mjs (a tiny, dependency-free ~580-line text module, first-paint-neutral) and then booting Pyodide on the same schedule as today. This ships green with no observable timing delta. An OPTIONAL improvement (behavior B), gated by a fresh test, is to defer the boot kick into the first-Run handler so BOTH engine.mjs and Pyodide load lazily on first ▶, genuinely eliminating engine/Pyodide cost from first paint — but this changes observable timing and risks any test assuming Pyodide warms during page load, so treat it as a separate test-gated follow-up, not part of the extraction. The four existing lazy gates (Automerge/ruff/JSZip/jsdiff) stay in index.html, untouched.

## Risks
- **Tests poke the Pyodide Python namespace directly (pyodide.runPython('_stop()'), _state['task'], _state['via_project'], id(_state['task']), and bare window.pyodide). If the extraction relocated the Python globals or hid pyodide behind the module, ~40 assertions across runmodel/spike-runstop/spike-pause/multifile go RED.** → Hard rule: engine.boot() installs BOOT_PY/PROJECT_PY into pyodide.globals EXACTLY as today (same runPythonAsync of the same strings) and the host sets window.pyodide from the engine's instance. The Python namespace is byte-identical post-boot. Run runmodel.mjs + spike-pause.mjs at every phase checkpoint.
- **String.raw escaping breakage: if the BOOT_PY/PROJECT_PY strings are reformatted, re-indented, or a `${` is introduced during the move, JS interpolation corrupts the Python source silently (compiles wrong or throws at runPythonAsync).** → Move verbatim; no reformatting. Add a build-free byte-diff check (strip only the const→export-const prefix) proving the strings are unchanged. Verified today the Python contains no `${` sequences — only bare `{}` literals, which String.raw passes through.
- **__yield__ / _pause_gate cooperative gate is the SINGLE chokepoint shared by both _start and _start_project (BOOT_PY comment line 868). Splitting the source across two exported consts (BOOT_PY, PROJECT_PY) and running them in two runPythonAsync calls must preserve the exact load ORDER — PROJECT_PY reuses _SyncBarrier/_is_gameloop/_Asyncify/_Awaiter/_InjectYield/_time_names/__yield__/__sleep__/_state/_stop from BOOT_PY.** → engine.boot() runs `await runPythonAsync(BOOT_PY); await runPythonAsync(PROJECT_PY)` in that strict order (same as lines 3373-3374). They are two strings in ONE module, so they can't drift out of sync the way two files could. spike-pause.mjs + subdirs.mjs gate the shared gate + the import machinery.
- **MetaPathFinder cooperative-import wrapper (_CoopPathFinder/_CoopLoader) anchors imports to _ROOT = os.getcwd() (the MEMFS cwd /home/pyodide) and is installed into sys.meta_path. It is sensitive to import order and to MEMFS state. If engine.boot() changes when/where getcwd is evaluated, project-origin detection (_is_project_origin) could misfire.** → _ROOT is captured at PROJECT_PY exec time (line 1231), which still happens inside engine.boot() at the same point in the boot sequence. Don't move PROJECT_PY exec before pyodide cwd is set. subdirs.mjs + multifile.mjs checks 2/3 (imported module with while True must stay cooperative) are the regression gate.
- **CORS / module-load on GitHub Pages: a wrong import specifier (e.g. '/vendor/engine.mjs' absolute, or a path that doesn't resolve under the /pygame-playground project subpath) 404s on Pages and the first Run throws.** → Use the SAME pattern as the working Automerge import: `import(new URL('./vendor/engine.mjs', document.baseURI).href)`. Relative + document.baseURI is already proven on Pages (line 3467). Commit vendor/engine.mjs to main (Pages serves committed files only; a .gitignore'd module would 404).
- **First-paint laziness regression: a careless P3 could make engine.mjs eagerly start Pyodide differently than today, or a static top-level import could eager-load the module graph, tripping shell.mjs / save.mjs check 5.** → Keep behavior A (eager boot kick stays in index.html, identical timing to today's line 3378). Use dynamic import() with a cached-promise gate, never a static import or <script type=module>. Leave the four existing lazy loaders in index.html untouched. shell.mjs + save.mjs check 5 gate this at every checkpoint.
- **AudioContext capture order: the Proxy wrapping of window.AudioContext (boot lines 3349-3356) MUST run BEFORE pygame.mixer loads. If moved into engine.boot() it could run later than today relative to other host init.** → Move it into engine.boot() as the very first step (it currently is the first thing boot() does, before loadPyodide). Order within engine.boot() is preserved 1:1. assets/audio paths aren't in the 16 batteries directly, but keep the step first to avoid silent audio breakage.
- **toPy/destroy memory hygiene: startProject does pyodide.toPy(files) + filesPy.destroy() and start.destroy()/startP.destroy() (lines 3404-3411). If the wrapper forgets a destroy(), Pyodide proxies leak across runs.** → Move the destroy() calls into the engine methods verbatim; they're part of the start/startProject method bodies. No new proxies introduced. multifile.mjs exercises repeated project runs and would surface a leak/crash.
