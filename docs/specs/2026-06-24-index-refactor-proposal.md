## Current state
The codebase is a fully-deployed single-file app (index.html ~3792 lines) with a redesigned UX (S1–S7 complete, 18/18 green tests verified). Core subsystems: (1) Collaboration CRDT (~1323 LOC, real-time sync via Automerge), (2) File operations + tree UI (~466 LOC, renderTabs is the 30+ call-site hub), (3) Project model (~464 LOC, files/order/entry/active + swapDoc persistence), (4) Embedded Python runtime (BOOT_PY + PROJECT_PY ~1437 LOC, AST-transformed loops, cooperative __yield__), (5) Asset filesystem (~227 LOC, MEMFS + IndexedDB triple-sync), (6) Auto-lint (82 LOC, ruff-wasm lazy), (7) History versioning (113 LOC), (8) Examples & promote-on-edit (93 LOC). Load-bearing seams: ONE CodeMirror (swapDoc-only entry), encode/decode boundary (CRDT keys vs human paths), runFile vs project.entry (editor file ≠ running file), roomOp guard (local-first, then mirror). Deep-refactor question (backlog #4) is deferred; Curriculum Phase 1 builds immediately on the redesigned platform without major restructuring."

## Drivers
- Single static index.html constraint (3792 lines) — no build step, no backend, GitHub Pages deployment
- First-paint lazy-load invariant (LOAD-BEARING TEST): zero network calls on boot → run path; Automerge/ruff-wasm/JSZip/jsdiff load only on first use
- ONE CodeMirror instance identity seam: project.files[active] === editor.getDoc() must always hold; swapDoc-only file switching (NEVER setValue)
- Cooperative async engine hidden from students: while True loops transformed to await __yield__() at frame boundaries; pause/resume via _pause_gate Event
- Flat MEMFS namespace (code + assets share bare names; organizational folders are UI-only, no runtime isolation)
- In-flight work collision: Slice B building (explorer action menu), Slice C queued (de-clutter), Curriculum Phase 1 starting (lesson loop) — all edit index.html
- Curriculum Phase 1 requirement: Demo/Tweak/Recreate/Verify phases for Lesson 0+0a/0b/0c without intelligent checking; substrate reuses Examples + split-run-model
- Test suite integrity: ~50 test call sites depend on #status tokens (exact strings), #runBtn, #stopBtn, #pauseBtn, #tabs/.tab[data-name], .CodeMirror selector; SEAM INVENTORY is the contract

## Options

### OPTION A: Internal Reorganization Only (No File/Architecture Changes)
Refactor index.html subsystems inline: split the ~3792 lines into named sections with clear comment boundaries, extract deeply-nested functions into top-level helpers (without changing globals or exported seams), consolidate repeated patterns (e.g., collision handling in project.add + assetFS.add into a shared uniquePath wrapper), restructure variable declarations into semantic groups. NO new files, NO build step, NO changes to window.* seams or test call sites. Trades: code readability + maintainability improve, but file stays monolithic (harder to parallelize dev, cognitive load on navigation).

*Single-file impact:* Stays single index.html. No impact.

**Pros:** Respects all non-negotiables: stays single static file, no build step, zero test churn; Lowest risk for in-flight work (Slice B/C can continue alongside without merge conflicts on subsystem boundaries); Fast to deliver (pure inline refactor, no seam rewiring); Lesson UI Phase 1 can start immediately (no architectural dependencies waiting); Existing test suite stays 100% green; no re-wiring of 50+ call sites

**Cons:** Monolithic file remains a cognitive bottleneck (~3800 lines in one editor tab); Parallel development stays difficult (all edits in index.html = contention); Doesn't address the deep-refactor question (backlog #4) — just postpones it; Collision-prone: future refactors will still struggle with the single-file constraint unless architecture changes; No modularization benefit for test reusability (e.g., a standalone engine.mjs for headless stepping isn't available)

### OPTION B: Multi-File Static ES Modules (NO Build Step, Relaxes Single-File Constraint)
Hand-author multiple .mjs files in vendor/ (e.g., vendor/app.mjs, vendor/collab.mjs, vendor/assets.mjs, vendor/history.mjs, vendor/engine.mjs) containing raw ES module code. index.html becomes a slim loader (~200 lines) that imports these modules dynamically via <script type=module> or inline import(). Load order managed via async/await or module-graph. Test seams (window.project, window.renderTabs, etc.) re-exported to window global scope from app.mjs, OR tests re-wired to import modules directly. Trades: cleaner code org + parallelizable dev, but breaks 'single-file' non-negotiable and requires careful module-boundary design to keep first-paint lazy-load invariant intact.

*Single-file impact:* Breaks single-file invariant (multiple .mjs files). Would require explicit relaxation of non-negotiable.

**Pros:** Cleaner code organization: each subsystem in its own .mjs (easier navigation, smaller files); Enables parallel development: separate branches can edit vendor/app.mjs and vendor/collab.mjs without index.html contention; Test seams can be per-module (clearer responsibilities); Headless stepping for curriculum Phase 2 behavioral harness becomes natural (import engine module, call step functions); Future bundling (if needed for performance) is a smaller leap (modules already exist, just add bundler)

**Cons:** VIOLATES 'single-file' non-negotiable (explicit in architecture map §0.12–16). Requires explicit sign-off to relax constraint.; Test seams must be re-wired: either all modules re-export to window (module-init boilerplate), or 50+ test sites re-import modules (test harness churn). Either way, ~50 test call sites need lockstep updates.; Lazy-load invariant must be preserved per-module: if app.mjs eagerly loads a heavy subsystem, first-paint breaks. Requires disciplined async imports (hard to enforce).; GitHub Pages still serves multiple files (no build step), but the multi-file structure is architecturally different. CI must verify all .mjs files are committed (or build breaks).; No bundler simplifies deployment but complicates module-graph control (no tree-shaking, no minification). Module-load order bugs are harder to debug.

### OPTION C: Bundler-Based Build Step (Breaks No-Build-Step Non-Negotiable) — ⚠ needs a build step
Introduce a Pages build action (GitHub Actions .github/workflows/build.yml) that runs esbuild/Vite/Rollup at publish time. TypeScript optional (could be pure JS). Bundles vendor/*.mjs + index.js into a minified single bundle.js. Supports tree-shaking (removes dead code), code splitting (lazy-load chunks), minification. Pages publishes the output bundles. Trades: modern tooling, smaller bundles, tree-shaking — but adds a build step to the deploy pipeline (requires CI/CD setup, slower deployments, new failure modes).

*Single-file impact:* Reduces to single index.html at deploy time (or index.js + chunks), but the source is now modular. Non-negotiable violation.

**Pros:** Smallest final bundle size (minification, tree-shaking); Modern dev experience (TypeScript if desired, dev/prod build configs); Code splitting: lazy-load heavy modules (Automerge, ruff, collab UI) as chunks that import dynamically; Can add source maps for debugging on GitHub Pages; Enables future frameworks (React/Svelte) if needed, though curriculum doesn't require it

**Cons:** VIOLATES 'no build step' non-negotiable (stated in architecture map §0.14). Requires explicit approval.; Adds GitHub Actions CI/CD complexity: build must run on every commit, times out or fails (breaks deploy). Human must debug CI, not just code.; Slower deployments (esbuild runs every push; ~30–60s overhead). Blocks hotfixes.; Build-artifact reproducibility risk: different esbuild versions or node versions produce different output (harder to audit diffs).; Test suite must run against the bundled output, not the source. Source-map debugging is required.; Slows onboarding: developer must set up Node.js + npm install before they can build locally.

### OPTION D: Hybrid — Hand-Authored Engine Module + Internal Refactor (Minimal Relaxation)
Extract ONLY the cooperative-run engine (BOOT_PY + PROJECT_PY + __yield__ + _state + _start/_start_project/_stop/_pause/_resume) into vendor/engine.mjs as a pure ES module that wraps pyodide calls. This module is referenced ONLY by run() in index.html (one call site). Everything else stays inline in index.html. App exports window.engine = { start, startProject, stop, pause, resume, stateSnapshot, taskDone } for lesson-UI integration. Trades: decouples the engine from the UI, enables headless stepping for Phase 2, but only one new file — minimal architecture change, easier to justify than full modularization.

*Single-file impact:* Adds one new vendor/engine.mjs file. Still stays single static index.html + helpers (similar to automerge-collab.mjs model). Minimal non-negotiable relaxation.

**Pros:** Focused extraction: only the most complex + reusable subsystem (cooperative engine) moves to .mjs; Curriculum Phase 2 behavioral harness gets a clean headless engine API (test/spike-pause.html model); Run control stays in index.html (no coupling risk), but engine is testable independently; Only ONE new file vendor/engine.mjs — minimal non-negotiable relaxation (can be argued as 'internal reorganization with one helper module'); Test harness can test engine stepping without loading full DOM (faster, clearer unit tests); Future lesson-UI integrations can import engine.mjs for custom runners (e.g., auto-check harness)

**Cons:** Requires moving ~250 lines of BOOT_PY + PROJECT_PY out of index.html. Must keep them as String.raw templates in the module (no change to runtime semantics).; Engine module must export the cooperative transform + __yield__ so lesson checks can run transformed code (adds export API design burden).; Test seams for engine shift: run() calls engine.start() instead of pyodide.runPython(_start), so tests that spy on pyodide.runPython lose that hook (must test via engine module instead).; Lazy-load sentinel must move: _amLoaded, _linter, _jszip, _diff stay in index.html; engine module can't gate them (engine is isolated). Could export readiness-check if needed.; Modest non-negotiable relaxation: one vendor/*.mjs file beyond automerge-collab.mjs. Requires approval but is easier to justify than full modularization.

## Load-bearing seams to preserve
- window.project model + project.files[active] === editor.getDoc() identity invariant (ONE CodeMirror). Any refactor that changes file-switching must preserve swapDoc-only entry (no setValue). Tests verify identity at startup + after load/add/setActive.
- renderTabs 30+ call sites + zero-arg contract. Refactor must preserve re-entrancy and keep the single window.renderTabs facade (even if logic moves to helpers, the entry point stays the same).
- #status textContent tokens (8 exact strings: 'starting…', 'loading Python…', 'loading pygame…', 'ready', 'running', 'finished', 'stopped', 'paused', 'error — see console', 'boot failed'). Every test gates these; friendly-error map depends on them. DO NOT reword.
- setStatus() single writer of #status (only function that writes .className and .textContent). If refactored, must preserve the signature + call sites.
- #runBtn, #stopBtn, #pauseBtn click targets (3 separate buttons, each with its own listener). ~50 test sites use page.click(). Keep ids stable or update tests in lockstep.
- Lazy-load gates for Automerge/ruff-wasm/JSZip/jsdiff (4 cached-promise loaders, one-shot each). Any refactor that hoists a loader to boot-time breaks first-paint invariant + shell.mjs smoke test.
- encode/decode boundary (encPath/decPath, lines 3516–3524): ONLY in Automerge doc keys are paths %2F-encoded. Locally all paths stay human. If collab-scope changes to multi-file, this seam must hold across file operations (mirrorMove/mirrorDelete).
- runFile global (set during run(), null when idle) vs project.entry (the designator, independent of shown editor file). Refactor must preserve this distinction; tests verify them separately (spike-runstop.mjs proof).
- project.load clears closure state (closedFolders, exampleOf, modifiedExamples, previewExample). Refactor must preserve the clearing order; if any subsystem caches these, it must be notified on load (or stale state leaks across project swaps).
- Collab applyingRemote flag (set during handle.on('change'), blocks editor.on('change') echo-back). Refactor must preserve the guard; if collaboration logic moves, the flag must follow (else remote ops echo, corrupting the doc).
- Asset atomicity (Slice A landed): assetFS.rename must re-key IndexedDB FIRST, then unlink MEMFS + update list. If refactored, order is LOAD-BEARING (quota failure mid-unlink = data loss). Keep the three-way sync (assetStore + MEMFS + list) atomic.
- Cooperative __yield__ as THE single frame-boundary chokepoint shared by both _start (single-file) and _start_project (multi-file). Refactor must preserve the shared gate; if pause/resume logic changes, it must affect both engines identically (else one mode breaks).
- Example promote-on-edit (S4 decided): first edit of a preview → adoptDoc into project.files, set exampleOf link, renderTabs marks as modified. Refactor must preserve swapDoc for preview + adoptDoc for promote (no setValue anywhere).
- Flat MEMFS + collision policy (file-extension-based routing .py→project.add / assets→assetFS.add; warn+suffix on dupe). Refactor must preserve both paths; if collision handling moves, keep the policy consistent (not split across subsystems).
- DOM id seams (15+ id-based selectors in tests): #status, #runBtn, #stopBtn, #pauseBtn, #tabs, #canvas, #console, #assetInput, #assetPanel, #historyPanel, #collabPanel, #examplesPanel, #saveBtn, #collabBtn, .CodeMirror, .tab[data-name], .hist-row[data-id], .remote-cursor, .remote-flag. If ANY id changes, 50+ test sites must update in lockstep.
- localStorage keys (PROJECT_KEY, legacy 'pygame-playground:code', history snapshots). Refactor must preserve key names; persistence format must not change (or saved projects won't load).
- Pyodide global (bare let pyodide at module scope, imported from CDN). Tests call pyodide.runPython, pyodide.globals.get, pyodide.FS.* directly. Refactor must preserve the entry point (or tests + lesson-UI can't access Python state).
